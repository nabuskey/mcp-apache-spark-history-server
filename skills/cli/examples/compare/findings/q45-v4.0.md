#### Plan Differences

The Comet plan has 60 nodes vs 70 in Default (-10 nodes), with 54 edges in both.

Key operator replacements:
- **Scans**: All 4 table scans (item x2, customer, customer_address, date_dim) replaced by CometNativeScan equivalents. 6 ColumnarToRow nodes reduced to 3, with 4 CometNativeColumnarToRow nodes added.
- **Joins**: Default uses 3 SortMergeJoin + 2 BroadcastHashJoin. Comet uses 1 SortMergeJoin + 1 CometSortMergeJoin + 3 BroadcastHashJoin. One SortMergeJoin was converted to a BroadcastHashJoin.
- **Exchanges**: Default has 7 Exchange nodes; Comet has 1 Exchange + 3 CometExchange + 2 CometColumnarExchange. The net exchange count drops from 7 to 6, but the critical change is that one exchange was eliminated entirely by converting a SortMergeJoin to a BroadcastHashJoin.
- **Sorts**: Default has 6 Sort nodes; Comet has 1 Sort + 3 CometSort. Total sort count drops from 6 to 4 because the join converted to BroadcastHashJoin no longer needs pre-sort on both sides.
- **Filters/Projects**: 7 Filter nodes reduced to 2 Filter + 5 CometFilter. 7 Project nodes reduced to 4 Project + 3 CometProject. 10 WholeStageCodegen nodes eliminated entirely.

The structural change is that the Comet plan converts one of the three SortMergeJoins into a BroadcastHashJoin, which eliminates an entire shuffle exchange round for the largest intermediate result set.

#### Root Cause

The +42.6% improvement is driven by two reinforcing factors:

1. **Join strategy change: SortMergeJoin to BroadcastHashJoin (primary factor, ~3.7s saved).** The Comet plan converts one of three SortMergeJoins into a BroadcastHashJoin. In Default, the second SortMergeJoin (stage 11941) must shuffle-write 1.5 GB of intermediate data, and the third SortMergeJoin (stage 11948) must shuffle-read that 1.5 GB. In Comet, the equivalent join (stage 6665) receives its build side via broadcast (360K records, 4.7 MB from stage 6657) and reads 0 B of shuffle. This eliminates 1.5 GB of shuffle write and 1.5 GB of shuffle read. The wall-clock savings come from removing stage 11941's 3.7s duration from the critical path -- Comet's stage 6660 (668ms) replaces Default's stage 11936 (792ms) and the broadcast delivery (76ms) replaces what was stage 11941 (3.7s). The additional BroadcastHashJoin exists because Comet's CometExchange and native scan operators change the cost model visible to AQE, allowing it to select a broadcast join where Default's AQE chose a sort-merge join.

2. **Columnar shuffle reducing data volume (secondary factor, ~1.4 GB less shuffle).** CometExchange and CometColumnarExchange use Arrow-based columnar serialization instead of row-based UnsafeRow serialization. For the first shuffle join, per-task shuffle read drops from 5.5 MB to 4.0 MB (-27%) and shuffle write drops from 4.0 MB to 2.8 MB (-30%) for the same 62M records. The customer scan shuffle write similarly drops from 316 MB to 232 MB (-27%) for 30M records. This reduces total shuffle I/O from 3.7 GB read + 3.7 GB write (Default) to 822 MB read + 1.8 GB write (Comet), with roughly 1.4 GB of the 2.9 GB read reduction attributable to the eliminated join and the remainder to columnar compactness.

3. **Reduced GC pressure (consequence).** GC time drops from 2.1s to 742ms (-65%), consistent with fewer shuffle materializations and fewer row-to-columnar conversions. Total task count drops from 3,272 to 725 (-78%) because the eliminated shuffle stages and their pre-sort stages no longer execute (Default has 19 skipped stages that still required scheduling overhead in prior AQE sub-plans, while Comet's plan has fewer stages per sub-plan to begin with).
