#### Plan Differences

Comet replaces the scan and pre-join processing for the `item` and `date_dim` dimension tables with native operators. The `catalog_sales` fact table scan and the SortMergeJoin remain on Spark's WholeStageCodegen path in both plans.

Operator replacements in Comet:
- `Scan parquet item` / `Scan parquet date_dim` replaced by `CometNativeScan`
- `Filter` on item and date_dim replaced by `CometFilter`
- `Project` on date_dim replaced by `CometProject`
- `Exchange` (hashpartitioning on item) replaced by `CometExchange`
- `Sort` (pre-join on item, post-agg sort for Window) replaced by `CometSort`
- `ColumnarToRow` replaced by `CometNativeColumnarToRow` at three transition points

Post-join aggregation and Window function exchanges are replaced:
- Two `Exchange` operators (nodes 12 and 8 in Default) replaced by `CometColumnarExchange` (nodes 13 and 8 in Comet)
- `Sort` for the Window function replaced by `CometSort`
- Four WholeStageCodegen blocks (6 through 9) eliminated, replaced by Comet native operators

The AQEShuffleRead after the first post-agg exchange coalesces 200 partitions into 10 in Default but into 23 in Comet, because Comet's larger shuffle output (25.1 MiB vs 10.7 MiB partition data) causes AQE to create more target partitions.

#### Root Cause

The regression is caused by CometColumnarExchange writing 2.3x more shuffle bytes than Spark's native Exchange for the post-aggregation reshuffle of 53,785 records. Default's Exchange (node 12) writes 10.2 MiB; Comet's CometColumnarExchange (node 13) writes 23.9 MiB for the same logical data (data size is identical at 10.0 MiB in both plans). This indicates the Arrow IPC columnar serialization format used by CometColumnarExchange has higher per-partition overhead when data is distributed across 200 shuffle partitions with small per-partition payloads (approximately 269 records per partition).

This shuffle write amplification has two downstream effects:

1. **AQE coalesces to more partitions**: The 25.1 MiB of partition data in Comet causes AQE to create 23 coalesced partitions instead of Default's 10. This alone adds 13 tasks to the query.

2. **Arrow decoding overhead in the reshuffle stage**: Each of Comet's 23 tasks in stage 704 spends p50 50ms on "decoding and decompression time" (1.2s total), reading from 4,600 blocks (vs 2,000 in Default). The per-task p50 is 82ms vs 53ms. Stage 704 takes 462ms vs Default's 94ms for stage 3319 -- a +368ms delta that is the dominant contributor to the overall +618ms regression.

Comet does recover approximately 374ms through faster native processing of the item dimension scan (stage 697 vs 3312: -319ms) and the catalog_sales scan (stage 696 vs 3311: -55ms), but this is outweighed by the +368ms in the reshuffle stage and +316ms in the main shuffle stage, plus +58ms in the final stage.

Net accounting of stage-level deltas:
- Stage 700 vs 3315: +316ms (shuffle write amplification in main stage)
- Stage 704 vs 3319: +368ms (Arrow decode overhead + more tasks from AQE)
- Stage 709 vs 3324: +58ms (CometNativeColumnarToRow transition overhead)
- Stage 696 vs 3311: -55ms (no Comet operators, normal variance)
- Stage 697 vs 3312: -319ms (CometNativeScan/CometFilter faster for item table)
- Stage 695 vs 3310: -68ms (broadcast, normal variance)
- **Total stage delta: +300ms** (matches reported stage time delta)
