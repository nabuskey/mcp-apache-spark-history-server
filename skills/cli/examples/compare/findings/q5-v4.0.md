#### Plan Differences

The Comet plan replaces several operators in the web_sales scan path with native Comet equivalents:

- **Scan parquet web_sales** (1,824 partitions, 2.16B rows): Replaced by `CometNativeScan` (node 109). The native scan reads 8.9 GB vs 11.2 GB for the same 2,159,930,761 rows.
- **Filter + Project + Exchange** on web_sales: Replaced by `CometFilter` (node 108), `CometProject` (node 107), and `CometExchange` (node 106). The CometExchange performs native columnar shuffle instead of Spark's row-based Exchange.
- **Sort** on web_sales stream in the join stage: Replaced by `CometSort` (node 105), which feeds into `CometNativeColumnarToRow` (node 104) before the SortMergeJoin.
- **Dimension table scans** (web_site, catalog_page, store, date_dim): All replaced by `CometNativeScan` + `CometFilter` + `CometNativeColumnarToRow`. These are broadcast tables and have negligible impact.
- **Final exchange** (node 6): Replaced by `CometColumnarExchange`, producing columnar shuffle output for the final aggregation stage.
- **ColumnarToRow**: Reduced from 11 to 7 instances; 5 replaced by `CometNativeColumnarToRow`.
- **WholeStageCodegen**: 8 codegen stages in Default (stages 16-24) are eliminated since the corresponding operators run natively in Comet.

The store_sales, store_returns, catalog_sales, and catalog_returns scan paths remain on Spark's native codegen (not replaced by Comet operators).

#### Root Cause

The 50.9% improvement (40.3s to 18.7s) is driven by two reinforcing effects, both originating from Comet's native processing of the web_sales path:

**1. Native columnar scan and shuffle for web_sales (stage 326/8043): 5.3s saved**

The web_sales table (2.16 billion rows, 1,824 partitions) is the largest table in this query. Comet replaces the entire scan-filter-project-exchange pipeline with native operators (CometNativeScan, CometFilter, CometProject, CometExchange). This produces three measurable effects: (a) input bytes drop from 11.2 GB to 8.9 GB because the native scan avoids deserializing columns not needed downstream, (b) shuffle write drops from 27.9 GB to 22.2 GB because the native columnar shuffle uses Arrow IPC encoding rather than UnsafeRow serialization, and (c) GC time drops from 7,236ms to 77ms because data remains off-heap in native memory. The stage completes in 5.4s vs 10.7s.

**2. Reduced shuffle read in the SortMergeJoin stage (stage 331/8048): 16.0s saved**

The downstream join stage reads the shuffled web_sales data. Default reads 27.9 GB of shuffle data; Comet reads only 2.2 GB. This 92% reduction occurs because the CometExchange wrote columnar Arrow data that the Comet shuffle reader can process without full deserialization. The CometSort (node 105) performs the pre-join sort natively on columnar batches, totaling 13.9 minutes of sort time across all tasks vs 31.9 minutes for Spark's Sort (node 107) -- a 56% reduction. The SortMergeJoin itself (still a Spark operator) benefits from receiving pre-sorted data more quickly. Median task duration drops from 13.4s to 5.2s, and the stage completes in 12.7s vs 28.7s.

**3. Cascading contention reduction on non-Comet stages: ~5s saved**

Stages 327 and 328 (store and catalog channels) use identical Spark-native operators in both plans and process identical data volumes. Yet they complete 4-5s faster under Comet (7.0s/6.2s vs 11.4s/11.6s). These stages run concurrently with the web_sales stages. Because Comet's native web_sales processing releases executor cores and memory sooner, the store and catalog stages face less resource contention and complete faster despite having no Comet operators themselves.

The combined effect: 5.3s (scan stage) + 16.0s (join stage) + ~5s (contention) accounts for the 21.6s wall-clock improvement. The join stage dominates because the 92% shuffle read reduction (27.9 GB to 2.2 GB) eliminates the primary bottleneck -- network transfer and deserialization of 2.16 billion shuffled rows.
