The plan node count drops from 27 to 22 (-5 nodes) due to elimination of `ColumnarToRow` and `WholeStageCodegen` wrappers.

#### Root Cause

The 322ms (46.5%) improvement on q41-v4.0 is driven by Comet's native columnar execution of the entire query pipeline. This query is a self-join on the `item` table (a relatively small dimension table with 360,000 rows), making it CPU-bound rather than I/O-bound.

The dominant factor is a 224ms reduction in the first scan+filter+aggregate stage (9288 vs 6609), which accounts for 69% of the total improvement. The second scan+join+aggregate stage contributes another 49ms (15%). Together these two stages account for 273ms, matching the aggregate stage time delta exactly (598ms - 325ms = 273ms).

Within these stages, Comet's native execution processes the same data faster at every task quantile. The max-task speedup (398ms to 174ms in the first stage, 160ms to 113ms in the second) is particularly significant because the skewed data distribution means the longest task determines stage completion time. The p50 improvement (108ms to 63ms, and 102ms to 63ms respectively) shows that even for near-empty partitions, Comet's native scan+filter path has lower overhead than Spark's ColumnarToRow + WholeStageCodegen pipeline.

The elimination of ColumnarToRow conversions is relevant: Default Spark reads Parquet into columnar batches, then converts to rows for WholeStageCodegen processing. Comet keeps data in columnar (Arrow) format throughout, avoiding this conversion. For a small table scanned end-to-end with filter+aggregate, the conversion overhead per row is a measurable fraction of total execution time.

The 18-task reduction (38 to 20) reflects that Comet's skipped stages return 404 and are not counted in the total, while Default's skipped stages are still counted. Both apps execute the same 20 actual tasks across 4 completed stages.

Shuffle bytes increase with Comet (28.8 KB to 99.1 KB) due to Arrow IPC columnar encoding overhead on very small record counts (168 and 354 records), but this has no meaningful impact on a query of this scale.
