#### Plan Differences

The query performs 5 independent full scans of `store_sales` (one per quantity-range bucket), each filtering, computing count/avg aggregates, exchanging partial results to a single reducer, then a final collect from `reason`. The plan has 5 identical scan-filter-aggregate-exchange pipelines.

**Operator replacements (per pipeline, x5):**
- `Scan parquet store_sales` replaced by `CometNativeScan parquet store_sales`
- `Filter` replaced by `CometFilter`
- `Project` replaced by `CometProject`
- `HashAggregate` (partial) replaced by `CometHashAggregate`
- `Exchange` replaced by `CometExchange`
- `ColumnarToRow` replaced by `CometNativeColumnarToRow`

**Structural change:** Each pipeline lost one node (-5 total across the plan, from 66 to 61 nodes). The `WholeStageCodegen (2)` wrapper around the scan-filter-project-partial-aggregate pipeline (5 occurrences) is eliminated because Comet replaces the entire pipeline with native operators. The final-aggregate `HashAggregate` (5 occurrences) and final `Project` (5 occurrences) remain as Spark JVM operators, receiving data via `CometNativeColumnarToRow`.

`Scan parquet reason` (1 occurrence) is replaced by `CometNativeScan parquet reason`.

#### Root Cause

The +41.5% improvement is driven by two reinforcing factors:

1. **Per-task compute speedup (primary).** Comet's native vectorized execution (CometNativeScan + CometFilter + CometProject + CometHashAggregate running as a fused pipeline in native code) reduces median task duration from 420-539ms to 276-361ms across the scan stages. This is a 33-48% reduction in per-task compute time. The query is purely CPU-bound: each of the 5 pipelines performs a full scan of store_sales (~8.6 billion rows), applies a range filter on `ss_quantity`, and computes count/avg aggregates. There is no join, no spill, and negligible shuffle. The native columnar execution processes these filter-and-aggregate operations on Arrow batches without row-level materialization or JVM object overhead.

2. **Near-elimination of GC overhead (secondary, amplifying).** Default Spark accumulates 27.3s of GC time across the 5 scan stages (averaging ~5.5s per stage), while Comet accumulates only 136ms total. The native execution path avoids JVM heap allocation for intermediate row data, keeping GC pressure negligible. In Default, GC pauses (~5s per stage) contribute to both per-task latency variance (max task times of 2.1-2.5s vs 1.6-1.8s in Comet) and overall stage duration by stalling executor threads.

The combination of faster per-task execution and eliminated GC pauses reduces total stage time from 5m35.6s to 3m51.9s (-1m43.8s). Because the 5 scan stages run concurrently on shared resources (115 cores across 23 executors), the wall-clock duration reduction is less than the total stage-time reduction. The critical path through the query -- the longest-running scan stage plus its reducer -- improved from roughly 69s to roughly 47s, matching the observed wall-clock delta of 21.4s.
