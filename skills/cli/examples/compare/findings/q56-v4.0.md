#### Plan Differences

The query unions results from three sub-queries joining store_sales, catalog_sales, and web_sales with customer_address and item, then aggregates by item description and color.

**Join strategy change (SortMergeJoin to BroadcastHashJoin):** In Default, each of the three sales tables (store_sales, catalog_sales, web_sales) joins with customer_address via SortMergeJoin, requiring Sort operators on both sides and a dedicated shuffle of customer_address (Exchange + Sort + ShuffleQueryStage). In Comet, AQE converts these three SortMergeJoins to BroadcastHashJoins. The customer_address table (703K rows, ~64 MB broadcast size as shown by BroadcastQueryStage statistics) is broadcast to all executors, eliminating the Sort operators and the shuffle-side exchange of customer_address.

**Comet native operator replacements:**
- date_dim scan/filter/project: CometNativeScan + CometFilter + CometProject + CometNativeColumnarToRow (vs Scan + ColumnarToRow + Filter + Project in Default)
- customer_address scan/filter/project: CometNativeScan + CometFilter + CometProject + CometExchange (vs Scan + ColumnarToRow + Filter + Exchange in Default)
- item semi-join: CometNativeScan + CometFilter + CometBroadcastHashJoin + CometBroadcastExchange + CometExchange (vs Scan + ColumnarToRow + Filter + BroadcastHashJoin LeftSemi + Exchange in Default)
- Shuffle exchanges: CometColumnarExchange replaces Exchange for the four partial-aggregation shuffles (three per-channel HashAggregate outputs + final aggregation)

**Stage count reduction:** Default has 33 stages; Comet has 14. The 19-stage reduction comes from eliminating the 6 Sort stages (2 per SortMergeJoin x 3 channels), the 3 shuffle stages for customer_address on the probe side, and various reuse/skip differences. Comet reuses the customer_address broadcast across all three sub-queries.

#### Root Cause

The 49.0% improvement (5.272s to 2.779s) is driven by AQE converting three SortMergeJoins to BroadcastHashJoins for the sales-table-to-customer_address joins. This conversion is enabled by the presence of Comet operators, specifically CometExchange on the customer_address shuffle, which provides AQE with accurate runtime statistics (sizeInBytes=2.7 MiB for 703K rows at ShuffleQueryStage 19 in Comet, vs sizeInBytes=10.7 MiB at ShuffleQueryStage 20 in Default). The smaller observed shuffle size (2.7 MB vs 10.7 MB) crossed below AQE's broadcast threshold, triggering the plan change.

This join strategy change produces two compounding effects:

1. **Elimination of sort and shuffle overhead:** The SortMergeJoin stages in Default required sorting both sides of each join across 200 shuffle partitions. The BroadcastHashJoin stages in Comet skip this entirely, reading directly from the original scan partitions (115/124/112 tasks). The combined stage time for these joins dropped from 8.559s to 1.434s (a 7.125s reduction in aggregate stage time, which translates to ~2.5s wall-clock savings given parallelism across executors).

2. **GC time reduction:** Default accumulated 2.823s of GC across all stages, with 1,217ms concentrated in the three SortMergeJoin stages alone (each holding 133-209 MB peak execution memory per task for sort buffers). Comet's BroadcastHashJoin stages recorded 0ms GC with 49-305 MB peak memory (hash table, not sort buffers). The total GC dropped to 206ms, a 2.617s reduction. The SortMergeJoin's per-task memory pattern (uniform 133-209 MB across all 200 tasks) indicates sort buffer allocation for the merge phase, which generates GC pressure as these buffers are allocated and freed.

The net wall-clock improvement of 2.493s is consistent with the join stage time savings (~7s aggregate reduced by executor parallelism to ~2.5s wall-clock) minus the slightly slower Comet scan stages (~0.3-0.4s slower per sales table scan, partially overlapping).
