#### Plan Differences

Comet replaces three small dimension table scan paths with native operators while leaving the large catalog_sales scan on the Spark codegen path:

- **date_dim scan** (1 task): `Scan parquet` + `ColumnarToRow` + `Filter` + `Project` replaced by `CometNativeScan` + `CometFilter` + `CometProject` + `CometNativeColumnarToRow`
- **item scan** (9 tasks, 2 occurrences): `Scan parquet` + `ColumnarToRow` + `Filter` + `Project` replaced by `CometNativeScan` + `CometFilter` + `CometProject` + `CometNativeColumnarToRow`
- **Aggregate subquery** (9 tasks): `ObjectHashAggregate` + `Exchange` replaced by `CometHashAggregate` + `CometExchange`, and the second `ObjectHashAggregate` by `CometHashAggregate`

The catalog_sales scan (118 tasks, ~947 MB) remains on Spark's WholeStageCodegen path in both apps. The main query structure (BroadcastHashJoin, final HashAggregate, final Exchange) is identical.

Node count decreased by 3 (55 to 52) because Comet's native scan combines `Scan` + `ColumnarToRow` into `CometNativeScan` + `CometNativeColumnarToRow` but removes intermediate WholeStageCodegen wrappers.

#### Root Cause

The -90.1% median regression is driven by extreme single-task stragglers in the Comet app, not by any systematic Comet operator overhead. The critical evidence:

1. **The regressing stages run identical Spark codegen, not Comet operators.** Both the first main stage (BroadcastHashJoin + HashAggregate + Exchange over catalog_sales, 118 tasks, ~947 MB) and the second main stage (final aggregate, 118 tasks) use Spark's WholeStageCodegen in both apps. Comet operators are confined to small broadcast dimension scans (date_dim: 1 task, item: 9 tasks) and a small aggregate subquery (9 tasks), which are actually faster under Comet.

2. **The p50 task durations are nearly identical between apps.** For the first main stage across all iterations, Default p50 ranges from 602-688ms and Comet p50 ranges from 651-714ms. The difference is negligible. The regression comes entirely from max task outliers.

3. **Comet iterations have more severe stragglers than Default.** In the Comet app, 3 out of 5 iterations have a max task that is 7-15x the p50, with the worst being exec 370 stage 6380 (max=7.072s, p50=676ms, ratio 10.5x) and exec 164 stage 1239 (max=4.585s, p50=301ms, ratio 15.2x). In the Default app, the worst straggler is exec 267 stage 3858 (max=1.848s, p50=298ms, ratio 6.2x). Default stragglers cap at ~1.8s while Comet stragglers reach 7.1s.

4. **Iteration variance confirms the straggler pattern.** Default duration range is 1,351ms (2,338-3,689ms) while Comet duration range is 8,856ms (2,132-10,988ms). Comet has 6.6x more variance. When no straggler occurs (exec 267 best Comet run, exec 576), Comet matches or slightly beats Default.

5. **The straggler occurs on a single task out of 118.** In every affected iteration, p75 is close to p50 and the max is a massive outlier, indicating exactly one task is delayed. This is consistent with infrastructure-level interference (e.g., co-located workload, network contention, or I/O scheduling) rather than a systematic algorithmic issue.

The regression is a measurement artifact caused by straggler tasks in the Comet app's large scan stages. These stages execute identical Spark codegen code in both apps. The Comet-specific operators (CometNativeScan, CometFilter, CometProject, CometHashAggregate, CometExchange) on the small dimension tables are actually faster than their Spark equivalents. The single-task outliers that inflate Comet's median are not attributable to any Comet code path, as they occur in stages that do not execute any Comet operators.
